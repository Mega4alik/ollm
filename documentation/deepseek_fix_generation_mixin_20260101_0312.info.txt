Summary of Changes:
Resolved a critical `AttributeError: 'super' object has no attribute 'generate'` in `DeepseekForCausalLM` and `DeepseekV3ForCausalLM` (Moonlight). This was caused by the custom model classes not inheriting from `transformers.generation.GenerationMixin`, which is required for the `generate()` method to work in newer versions of the Transformers library (v4.50+).

Detailed Implementation Notes:
1. Updated `src/ollm/modeling_deepseek.py`:
   - Imported `GenerationMixin` from `transformers.generation`.
   - Updated `DeepseekForCausalLM` class definition: `class DeepseekForCausalLM(DeepseekPreTrainedModel, GenerationMixin):`.

2. Updated `src/ollm/modeling_moonlight.py`:
   - Imported `GenerationMixin` from `transformers.generation`.
   - Updated `DeepseekV3ForCausalLM` class definition: `class DeepseekV3ForCausalLM(DeepseekV3PreTrainedModel, GenerationMixin):`.

3. Verified other models:
   - Checked `mistral.py`, `mixtral.py`, `qwen2.py`, `qwen3.py`, `qwen3_next.py`. These rely on standard `transformers` classes (e.g., `MistralForCausalLM`) which already include the necessary mixins, so no changes were needed for them.

Fixes & Caveats:
- Fix: The `generate()` method now works correctly for DeepSeek and Moonlight models.
- Caveat (Size [0] Warnings): Users may see `torch.Size([0])` mismatch warnings during model loading (e.g., `model.layers.27.mlp.experts.9.gate_proj.weight`). This is a known side-effect of the project's custom "load-compute-unload" strategy where layers are aggressively offloaded (replaced with meta-tensors) during initialization to save memory. These warnings are generally non-fatal if the custom loader (`loader.load_dict_to_cuda`) is working correctly during inference.
- Hardware: No specific hardware changes, but this fix ensures the code runs on the intended hardware (e.g., Blackwell nodes) without crashing.

Usage Instructions:
No changes to usage. Run the example script as before:
```bash
python3 scripts/example_deepseek_moe.py
```
Ensure dependencies are installed.
